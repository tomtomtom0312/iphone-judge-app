# 越境販売 判断アプリ — ROADMAP（引き継ぎ資料）

不用品回収の現場で写真を撮り、その場で「売る or 処分」と「どの販路で売るか」を即判断するための iPhone 向け Web アプリ。

最終更新: 2026-06-11

> **ドキュメント運用**: 本ファイル `ROADMAP.md` が長期計画の**正本**。大きな機能追加・方針変更があったら更新する。
> 直近の作業状況は別ファイル `HANDOFF.md`（300〜500字の軽量な引き継ぎメモ）で管理する。

---

## 最終ゴール（実現したい体験）

```
写真撮影
  ↓
AI商品認識（機種・状態を自動推定）
  ↓
eBay相場取得（想定売価を自動入力）
  ↓
Shopee相場取得（第2販路の売価を自動入力）
  ↓
利益計算
  ↓
推奨販路表示（どの販路が一番儲かるかを提示）
```

---

## 設計の大原則

**判定エンジンは不変。「入力欄をどう埋めるか」だけを段階的に差し替える。**

アプリの中核は「機種・状態・売価・手数料・送料 → 利益計算 → 判定」。
この計算ロジックは最後まで変えない。各フェーズで増えるのは入力の埋め方だけ。

```
              ┌─ Phase1: 人が手入力
入力欄  ←──────┼─ Phase2: AIが機種/状態を自動入力（人が上書き可）
              ├─ Phase3: eBay相場が売価を自動入力
              └─ Phase4: Shopee相場が第2販路を自動入力
                    ↓（どの埋め方でも）
              判定エンジン（不変）→ 推奨販路表示
```

- **人による上書きを常に最優先**にする。AI や相場取得が外しても現場は止まらない。
- 販路（eBay / Shopee …）は `MARKETPLACES` レジストリで定義済み。販路追加は1ブロック足すだけ。
- 構成の転換点は **Phase2（静的 → 最小バックエンド）の一度だけ**。それまでは静的1ファイルを維持する。

## フェーズ計画

### Phase 1 — 静的 eBay 版を「現場で使えるレベル」に仕上げる（← 現在ここ）

構成: 静的 HTML 1ファイル・サーバー不要・全データは端末の localStorage。
目的: 毎日使っても困らない信頼性。AI に進む前に土台を固める。

| # | 項目 | 状態 |
|---|------|------|
| 1 | 状態選択の改善（「新品」「未使用品」を追加） | ✅ 完了 |
| 2 | 利益率を判定に追加（利益額＋利益率の複合判定） | ✅ 完了 |
| 3 | 送料編集（発送方法ごとの送料を編集可能に・販路ごと保存・初期値に戻す） | ✅ 完了 |
| 4 | CSV 出力（履歴を書き出して後で集計。UTF-8 BOM付き・CRLF） | ✅ 完了 |
| 5 | 現場テスト（実機で使い込み、改善点を洗い出す） | ⬜ 未着手 |

補足: CSV 用に「商品名/メモ」入力欄を追加（履歴表示・CSV列に反映）。

#### 判定ロジック（現行仕様）
```
売上(円)   = 想定売価(通貨) × 為替レート
販売手数料 = 売上 × 手数料率
利益(円)   = 売上 − 手数料 − 原価 − 送料

見送り : 利益額 < 見送りライン  または  利益率 < 見送り利益率
推奨   : 利益額 ≥ 推奨ライン   かつ   利益率 ≥ 推奨利益率
要検討 : それ以外
```
為替・手数料・各しきい値は **販路ごとに** 設定保存。eBay 既定値: 為替155円/USD・手数料16%・推奨3,000円/20%・見送り0円/0%。

### Phase 2 — AI 商品認識版（★最初のバックエンド導入点）

目的: 写真から機種・状態を自動推定し、入力欄に初期値としてセット（人が上書き可）。

> **現場テストでの裏付け（2026-06-11）**: 「商品名を手入力したくない／写真から商品名を推定したい」という要望が出ており、本フェーズの需要を確認済み。Phase 2 で最優先に応えるべき機能。

- フロントは静的のまま、**サーバーレス関数を1つだけ**追加（API キー秘匿のため必須。Vercel/Cloudflare Functions 等。APIキーをブラウザJSに置くのは厳禁）。
- 写真 → Claude Vision（`claude-opus-4-8`）→ 構造化出力で `{商品名, 機種, 状態, 信頼度}` を JSON 固定取得 → 入力欄に自動セット。
- AI 結果はあくまで初期値。信頼度が低い項目は警告し手動入力を促す。
- 構成転換点。複雑化を避けるため処理は1関数に閉じ込める。

#### コスト試算（2026-06 時点の料金・為替155円換算）
写真1枚 ≈ 画像入力1,500〜2,000トークン + 出力150トークン程度。
| モデル | 単価(入力/出力 per 1M) | 1枚 | 3,000枚/月 |
|---|---|---|---|
| Opus 4.8 (`claude-opus-4-8`) | $5 / $25 | 約1.5〜2円 | 約4,500〜6,000円 |
| Sonnet 4.6 (`claude-sonnet-4-6`) | $3 / $15 | 約1円 | 約3,000円 |
| Haiku 4.5 (`claude-haiku-4-5`) | $1 / $5 | 約0.4円 | 約1,200円 |
方針: まず Opus 4.8 で精度確認 → 十分なら Sonnet/Haiku に下げてコスト最適化。

#### Phase 2 MVP の状態（2026-06-11 / 2026-06-12 拡張）
- 実装済み・未デプロイ。`api/recognize.js` + `package.json` + フロント配線。売価取得は引き続きなし。
- **認識スキーマをカテゴリ別に拡張（2026-06-12）**: `{productName, category, brand, series, setName, cardName, boxName, storage, color, condition, estimatedRank, confidence, notes}`。`category` enum = `iPhone / trading_card_single / trading_card_box / figure / other`。カテゴリに関係する項目だけを埋め、無関係は空文字。`max_tokens:400`。
- **連続撮影モード（2026-06-12）**: 撮影画面に「単品判定／連続撮影」切替を追加。連続モードは1枚ごとに撮影→クロップ調整→「この位置でAI認識」→ドラフト履歴へ保存（売価0=未計算）。売価・原価・送料・メモは後から各履歴の「編集」で入力し、利益を再計算。`categoryHint`（任意）で同種まとめ撮りのカテゴリをバイアス可能。
- **撮影後クロップ（2026-06-12）**: 重さ回避のため `getUserMedia` のライブカメラは採用せず、撮影/選択した**静止画**に対してプレビュー上で位置・拡大の調整を行う方式。点線ガイド枠（枠外は薄暗）＋指ドラッグのパン＋＋/−/リセットのズーム（ピンチ不使用）。AIへは枠内を canvas で切り出して送る（履歴400px・AI768px）。作業画像は最大辺2000pxへ縮小。リアルタイムプレビューは将来も避ける方針。
- 前提: 現場のモバイル回線あり（ケースA）。電波が無い現場向けの「オフライン撮りため→後でまとめて認識」は将来対応。
- タイムアウト: フロント30秒（AbortController）・Vercel関数30秒（maxDuration）。

#### 認識対象の方針（中央の主役だけを見る）
現場はゴミ屋敷のように周囲に物が多い前提。Googleレンズのように対象商品だけを認識する。
- プロンプトで「背景・周辺物（ゴミ袋/家具/段ボール等）は無視し、中央に最も大きく写っている1点だけを特定」と明示。
- 複数商品が写る場合も最も中央で大きい1点のみ対象。判別困難なら confidence を低く。
- 撮影画面に「商品を中央に大きく」案内、プレビューにガイド枠（視覚的目安・トリミングはしない）。
- **トリミング（中央を実際に切り出して送る）は Phase 2.5 以降で検討。** MVPはプロンプト指示＋撮影誘導で対応。

#### Phase 2 残課題（本格運用前に対応）
- **関数URL公開リスク**: `/api/recognize` は誰でも叩け、第三者が API 枠を消費し得る。当面は Anthropic の **Spend limit** を保険に運用し、本格運用時に**共有トークン等の簡易認証**を関数に追加する。
- 信頼度が低い結果のUI強調、撮り直し導線の改善。

検討メモ: 代替案として端末内ML（オフラインだが精度・準備が重い）、バーコード/IMEI読取（箱がある新品向け）。中古主体のため当面はクラウド Vision を推奨。

### Phase 3 — eBay 相場自動取得

目的: 想定売価を自動入力して判定精度を上げる。

- Phase 2 のバックエンドに相場取得を相乗り（新インフラ不要）。
- eBay API または売却済み相場の参照 → 売価欄を自動入力。
- AI 機種特定（Phase 2）と直結し「写真 → 機種 → 相場 → 判定」が一気通貫に。

---

### Phase 4 — Shopee 対応＝推奨販路表示の完成

目的: 複数販路を横並び比較し、最も儲かる販路を提示。

- `MARKETPLACES` に Shopee を追加（構造は対応済み。`index.html` 内にコメントで雛形あり）。
- Shopee 固有の検討点: 手数料が販売手数料＋決済手数料＋キャンペーン手数料の複合・国ごとに率が違う。
  → `fee` を単一率でなく内訳を持てる形に拡張するのが本実装時の論点。
- 各販路で利益を計算 → 「eBay ¥3,200 / Shopee ¥2,800 → eBay 推奨」のように比較表示。
- これで最終ゴールの「推奨販路表示」が完成。

---

## 進め方の方針

- 順序は **静的版 → AI認識版 → eBay相場取得 → Shopee対応** で固定。
- **Phase 1 を実機で使い込んでから Phase 2 へ進む。** バックエンド導入は運用負荷が増えるため、
  「手入力のどこが本当に面倒か」が現場で見えてから着手する。
- MVP 段階のため、機能を増やしすぎて複雑化させない。各フェーズは単体で使える状態を保つ。

## 技術メモ

- 現状: 単一 `index.html`（HTML/CSS/バニラ JS）。ビルド・依存・サーバーなし。
- データ保存: ブラウザ localStorage。
  - `judgeActiveMarket` … 選択中の販路
  - `judgeSettings_<販路ID>` … 販路ごとの為替・手数料・しきい値
  - `judgeShipping_<販路ID>` … 販路ごとの送料の編集値（発送方法labelをキーに上書き）
  - `judgeHistory` … 判定履歴（写真サムネイル含む）。各エントリに `category`/`estimatedRank`/`ai`（認識生フィールド）/`draft`（売価未入力フラグ）を保存。`draft:true` は売価0で利益未計算、編集で売価を入れると確定。
- 写真は保存時に最大幅400px・JPEG品質0.7のサムネイルへ縮小（localStorage 節約）。
- ローカル確認: `python3 -m http.server 8000` → iPhone Safari で `http://<PCのLAN IP>:8000`。
