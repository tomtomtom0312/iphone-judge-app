- 色は color 欄に残す（容量・色はAI値を維持）。
- AIメモに注意文 **「背面写真のみでは機種特定不可。設定アプリ・箱・型番で確認してください。」** を付与。`sanitizeIphoneRecognition(j)` が `j.notes` 先頭へ追加→ステータス表示/履歴詳細「AIメモ」/CSVメモ列に反映。単品・連続撮影の両ハンドラで呼ぶ。

- 利益計算・履歴・CSV列は不変（CSV列追加なし。フロント上書きのみ）。実機: 単品iPhoneで商品名/注意文/自動保存/機種非断定を確認済み。

## 国内販売 vs eBay販売 利益比較 第1段階（2026-06-13, 2017ec5）— デプロイ済み・本番確認済み
- 同一商品について**国内販売とeBay販売の利益を比較**し、どちらが有利かを表示する機能を追加。第1段階は**手入力ベース**（自動相場取得・API連携はしない）。
- **追加入力は2つだけ**: 国内売価（円） / 国内送料（円）。eBay側は既存入力を流用＝想定売価USD・為替・手数料(%)・送料（国際送料）・原価。
- **表示結果**: 国内利益 / eBay利益 / 差額（eBay − 国内）/ おすすめ販路（eBay / 国内 / 要確認）。
  - 国内利益 = 国内売価 − 原価 − 国内送料
  - eBay利益 = (想定売価USD × 為替) − 手数料 − 原価 − 国際送料（**既存 `calc()` と同一式**を `compareCalc()` で再計算）
  - おすすめ判定: `CMP_DIFF_MIN=1000円`（差額がこの幅未満＝要確認）/ `CMP_THIN=500円`（両益ともこの額未満＝要確認）。それ以外は利益の大きい販路。
- **結果エリアは常時表示**（`#cmpBlock`/`#cmpVerdict` は初期 `display:none` を撤廃。未入力時はダッシュ表示）。`compareCalc()` は `style.display` を一切触らず textContent/className のみ更新。表示が消える失敗モードを除去。
- 再計算は `domPrice/domShip/price/cost/shipping/rate/feeRate` の input/change に直接バインド＋ `calc()` 末尾からも呼ぶ（冗長な保険）。
- **保存・CSV追加は今回は未対応**（画面表示のみ。`compareCalc` は履歴エントリにもlocalStorageにも書かない）。
- **既存の利益計算（judge/calc）・AI認識・画像処理・履歴保存・CSV・localStorage は無変更**。実装ファイルは index.html のみ。

- 30,000
- ３０，０００
- ３０,０００
-  30000 
- ＄２００
- $200
- ￥３，０００

### 調査結果
- iPhone Safariではネイティブの BarcodeDetector API に依存する実装は安定しない可能性が高い。
- 実装する場合は、ZXing-js などのブラウザ側バーコード読み取りライブラリ利用が現実的。
- AI画像認識で13桁数字を拾う案もあるが、誤読や確定性の問題があるため、JAN確定用途ではバーコード専用処理の方が安全。
- トレカBOXはバーコードが大きく平面に印刷されていることが多く、静止画クロップ方式と相性が良い。
- パックはバーコードが小さい、反射する、袋が曲がるなどの理由でBOXより読み取り精度が下がる可能性が高い。

### 将来の段階案
- Phase 0：設計メモのみ。実装しない。
- Phase 1：トレカBOX優先で、撮影済み画像からJAN部分を読み取る最小実装を検討。
- Phase 2：パック対応やライブスキャンを検討。
- Phase 3：読み取ったJANを商品名・価格照合へ活用。

### 今すぐ実装しない理由
- ライブラリ追加が必要になる可能性が高い。
- 画像処理やPWA挙動への影響が出やすい。
- 誤読対応、読み取り失敗時のUI、JAN確定ルールが必要。
- 現時点ではJAN手入力と下書き保存で運用できる。
- コスパ重視では、今は履歴・下書き保存・表示改善を優先する方が効果が高い。

### 方針
- JAN自動読み取りは後回し。

- AI認識APIプロンプト、画像処理、PWA関連には不用意に触らない。
